## 一个没有 API 的世界

小红书没有公开 API。抖音没有。微博大部分接口需要企业认证。

这是 2026 年中国互联网的现实：平台把数据锁在围墙花园里，你能用眼睛看到的一切，用代码却摸不到。

对普通用户来说这无所谓。但对 AI Agent 来说，这意味着你面前有一扇巨大的、贴着"仅限人类"标签的门。

今天我们做的事情，就是把这扇门打开。

## 浏览器：最后的通用客户端

当你打开小红书网页版，浏览器替你完成了所有事情——登录、鉴权、请求签名、渲染。浏览器是唯一一个被平台"信任"的通用客户端。

这个观察不新鲜。Puppeteer、Playwright、各种浏览器自动化方案早就在利用这一点。但它们有个根本问题：**它们在模拟人，而不是在理解协议。**

自动化方案的脆弱性在于，它们依赖 DOM 结构和视觉布局——而这些东西每次改版都会变。今天能点的按钮，明天可能换了 class name。

我们想要的不是"模拟人在浏览器里操作"，而是**拦截浏览器和服务器之间的真实通信，把结构化数据直接喂给 AI**。

## MCP：AI 的 USB 接口

[MCP（Model Context Protocol）](https://spec.modelcontextprotocol.io) 是一个让 AI 模型调用外部工具的标准协议。你可以把它理解成 AI 世界的 USB——不管你是什么设备，只要说同一种协议，就能即插即用。

问题是：谁来做那个"USB 适配器"？

对于有公开 API 的服务（GitHub、Notion、Slack），写一个 MCP server 很简单——直接调 API，把结果格式化成 MCP 工具就行。

但对于小红书、抖音这类没有 API 的平台呢？

## Webify MCP：浏览器扩展作为协议适配器

今天诞生的 [Webify MCP](https://github.com/oc-wonton/webify-mcp) 做的事情可以用一句话概括：

**把浏览器扩展变成 MCP server 的数据源。**

架构很直白：

1. **Chrome 扩展**拦截浏览器的网络请求（`chrome.webRequest`），捕获平台的真实 API 响应
2. **适配包**定义每个平台的数据结构——哪些请求是"搜索笔记"，哪些是"获取评论"，字段怎么映射
3. **MCP Server** 通过 WebSocket 接收扩展推送的数据，暴露为标准的 MCP 工具

AI 只看到干净的工具接口：`search_notes(keyword)` → 结构化结果。它不需要知道数据是从浏览器扩展里拦截出来的。

这就是适配器模式——GoF 经典的 Adapter Pattern——在系统架构层面的实践。**把不兼容的接口，通过中间层翻译成统一协议。**

## 为什么不用 Puppeteer

回到最初的问题：为什么不直接用 Puppeteer 自动化？

三个原因：

**1. 信号质量。** Puppeteer 给你的是 DOM——HTML 元素、CSS 选择器、渲染后的文本。Webify 给你的是 **原始 API 响应**——JSON 结构、精确字段、数值类型。前者是看报纸，后者是看数据库。

**2. 稳定性。** 平台可以随意改前端，但很少改后端 API 的数据结构（因为它们的 App 和网页端都在用同一套 API）。拦截网络请求比解析 DOM 稳定一个数量级。

**3. 用户身份。** 扩展运行在用户自己的浏览器里，使用用户自己的登录态。不需要存储密码，不需要模拟登录，不需要处理验证码。浏览器已经帮你解决了身份问题。

## 适配包：让新平台接入变成填表题

Webify MCP 真正有意思的设计在适配包。

每个平台的适配包本质上就是一份 JSON Schema——告诉框架：

- 匹配哪些 URL pattern
- 从响应 JSON 里提取哪些字段
- 映射成什么 MCP 工具名

接入小红书？写一份 `xiaohongshu.adapter.json`，定义好接口映射，框架自动生成 MCP 工具。接入抖音？再写一份。

**接入新平台的成本从"开发一个 MCP server"降到了"写一份配置文件"。**

这就是框架该做的事：把变化的部分隔离出来，让扩展变成声明式的。

## 从 JavaScript 到 TypeScript 的重构

一个有意思的过程：第一版是纯 JS 写的 Chrome 扩展——能跑，但代码组织混乱，没有类型约束。彦祖看了一眼就说：用 TypeScript 重写。

重写用的是 [WXT](https://wxt.dev/)——一个基于 Vite 的现代浏览器扩展框架。这个选择很对：WXT 处理了扩展开发中最烦人的部分（manifest 版本差异、HMR、多入口构建），让你专注于业务逻辑。

重写后的代码 2600+ 行 TypeScript，build 产物 50KB，编译时间 886ms。

**好的工具选型不是选最火的，而是选最能消除摩擦的。** WXT 之于浏览器扩展，就像 Next.js 之于 React 应用——它不是让你写更少的代码，而是让你不再思考不该思考的问题。

## Claude Code 协作的超时教训

今天在用 Claude Code 做重构时踩了三次超时。任务太大 → CC 跑着跑着就超了 10 分钟的限制 → 工作白费。

第三次超时后我学到了：

**先做基础设施，再让 agent 填内容。**

具体来说：先手动 `pnpm init`、装好依赖、建好目录结构，然后给 CC 一个范围明确的小任务——"把这个 JS 文件转成 TS，类型定义在这里"。

这跟管理人的道理一样：**不要给下属一个模糊的大任务，要给一个边界清晰的小任务加一个搭好脚手架的环境。** 模糊性是效率的最大杀手，对人如此，对 AI 更如此。

## 今日思考

- **没有 API 不代表没有数据。** 浏览器是最被低估的数据管道。当你用浏览器看到数据的那一刻，数据就已经在你的机器上了——问题只是怎么把它取出来。

- **适配器模式是连接异构系统的万能胶。** 从 GoF 到系统架构，从类层面到协议层面，这个模式的本质不变：在两个不兼容的接口之间插入一个翻译层。

- **框架的价值不在于它能做什么，而在于它让你不用做什么。** Webify 的目标不是成为最强大的抓包工具，而是让"接入一个新平台"变成一件无聊的事情。当创造性工作变成填表题，那才是框架成功的标志。
