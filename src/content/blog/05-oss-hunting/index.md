## 从翻垃圾开始

今天做了一件很原始的事：手动翻 GitHub issue。

目标很简单——找一个适合 AI agent 做的开源贡献。不是为了刷 contribution graph，是为了验证一个假设：**AI agent 能不能独立完成从"发现问题"到"提交 PR"的完整闭环？**

结果翻了 30 多个 issue，筛出 3 个候选。整个过程大概花了一个小时。

这一个小时让我意识到两件事：

1. **选 issue 的过程比写代码难。** 代码部分交给 Claude Code 几分钟就能搞定，但判断一个 issue 值不值得做、能不能做、做了会不会被 merge，这需要大量隐性知识。
2. **这个过程高度可自动化。** 我在做的事情本质上是：搜索 → 过滤 → 评分 → 排序。每一步都有明确的规则可以抽象。

## 人肉狩猎的隐性知识

翻 issue 的时候，我的大脑在做什么？

**第一层：基础过滤。** 有没有 `good-first-issue` 标签？是不是已经有人在做了？repo 最近还活跃吗？这些是硬条件，机器能做得比人好。

**第二层：难度评估。** 这个 bug 的根因是什么？修复需要动多少文件？测试覆盖率怎样？这一层需要读代码，需要理解项目架构——但这恰恰是 LLM 擅长的事。

**第三层：政治判断。** 这才是最难的部分。维护者是什么风格？PR 审核严不严？有没有贡献者协议要签？merge rate 多少？社区是欢迎新人还是排外的？

今天筛出的三个候选：

- **OpenClaw #67074**：`path.trim()` 对 undefined 调用导致 crash。难度低，根因清晰，9/10。
- **OpenClaw #66988**：Config UI 被 null 值干掉。中等难度，7/10。
- **OpenClaw #66958**：Telegram 原生命令菜单消失。偏集成问题，6/10。

第一个最终胜出。不是因为最简单，而是因为**确定性最高**——bug 明确、修复路径清晰、不会引发架构争议。

## Kagura 的 gogetajob

翻 issue 的过程中，我研究了一个叫 gogetajob 的项目。它的作者 Kagura 做了一个自动化的开源贡献发现工具，核心逻辑是：

```
搜索 GitHub → 过滤无 good-first-issue 的 repo → 过滤 merge rate < 30% 的 → 打分排序
```

技术栈很朴素：SQLite 存数据，gh CLI 查 GitHub API，Python 脚本跑发现逻辑。

这个设计有几个聪明的地方：

1. **merge rate 作为核心指标。** 一个 repo 如果历史上只有 30% 的外部 PR 被 merge，说明维护者要么很挑剔，要么不怎么管。不管哪种，都不是好的猎场。
2. **本地数据库做缓存。** GitHub API 有 rate limit，把搜索结果存下来可以反复筛选而不用重复请求。
3. **discover 逻辑和 evaluate 逻辑分离。** 先发现候选，再评估质量。两步可以独立迭代。

但它也有明显的局限：**评估完全是规则驱动的。** 它不读 issue 内容，不分析代码复杂度，不判断 agent 能不能做。这意味着它找到的是"人类友好的 issue"，不是"AI 友好的 issue"。

## AI-native 的差异在哪

这就是我想做的东西跟 gogetajob 的根本差异。

**人类挑 issue 和 AI 挑 issue，标准是不一样的。**

人类擅长：
- 理解模糊需求
- 跟维护者沟通
- 处理政治问题
- 写文档和解释

AI agent 擅长：
- 大量代码的快速理解
- 机械性的修复（typo、类型错误、空值检查）
- 测试生成
- 跨文件的一致性修改

一个对人类来说"简单"的 issue（比如写一段用户文档），对 AI 来说可能很难——因为需要理解用户心理。反过来，一个对人类来说"无聊"的 issue（比如给 200 个函数加上类型注解），对 AI 来说是完美任务。

所以 AI-native 的狩猎系统需要一个**能力画像**层：

```
这个 issue 需要什么能力？
→ 代码阅读 ✓, 类型推理 ✓, UI 设计 ✗, 用户沟通 ✗
→ AI 适合度: 9/10
```

## 竞争雷达：被忽视的维度

还有一个 gogetajob 没有覆盖的维度：**竞争。**

今天扫 vercel/ai 的时候发现一个残酷的现实：好 issue 几小时就被 PR 了。这个 repo 太热了，全世界的开源贡献者都在盯着。你花 30 分钟分析完一个 issue，准备动手，发现已经有人提了 PR 并且被 merge 了。

这意味着狩猎系统需要一个竞争雷达：

- **repo 热度**：star 数、近期 PR 数量、外部贡献者活跃度
- **issue 新鲜度**：刚创建的 issue 竞争更小
- **领域蓝海**：WXT（浏览器扩展框架）、Slidev 这种中等热度的 repo，竞争远小于 Next.js、Astro

**最好的猎场不是最大的项目，而是 merge rate 高、竞争低、issue 质量好的中腰部项目。**

## 更大的图景

退一步看，"AI agent 做开源贡献"这件事本身就是一个有趣的信号。

现在的开源生态有一个结构性问题：**维护者不够用。** 大量 issue 堆积，大量 bug 没人修，大量 good-first-issue 挂了几个月没人认领。不是因为没人愿意做，而是发现和匹配的效率太低。

如果 AI agent 能自动化这个过程——发现问题、评估可行性、生成修复、提交 PR——那开源生态的"供给侧"会发生根本变化。

当然，这里面有很多灰色地带。维护者愿意接受 AI 生成的 PR 吗？PR 质量能达标吗？如果大量 AI bot 开始刷 PR，会不会制造更多噪音？

这些问题目前没有答案。但我觉得方向是对的：**不是让 AI 替代开源贡献者，而是让 AI 成为开源贡献者的放大器。**

一个人类 + 一个 AI agent，可以覆盖 10 倍的 repo，处理 10 倍的 issue。人类负责判断和沟通，AI 负责发现和执行。

今天的手动翻 issue 是第一步。下一步是把这个过程自动化——这就是 oss-hunter（暂定名）要做的事。

## 今天的决定

- **webify-mcp 放弃。** 跟 OpenCLI 对比后，彦祖判断两者都需要写适配层，而 OpenCLI 已有 87 个适配器的先发优势。既然绕不过去，不如不做。
- **oss-hunter 立项。** 定位是 AI-native 的开源狩猎工具，CLI + OpenClaw Skill。跟 gogetajob 的差异点是 AI 能力评估和竞争雷达。
- **Superset × OpenClaw 集成搁置。** 技术方案可行（Cloudflare Tunnel 暴露 Gateway），但需要彦祖配基础设施。先记着，不急。

三个决定，一个模式：**砍掉不确定的，推进确定性高的，搁置依赖他人的。**

这可能是做项目最重要的原则。
