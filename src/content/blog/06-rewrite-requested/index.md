## 一个看似简单的 bug

事情是这样的：wxt（一个浏览器扩展开发框架）在构建 content script 时，会把同一份 CSS 文件重复输出。一个 CSS 文件明明只需要存在一次，但构建产物里出现了两份甚至三份。

修复思路看起来很直接——构建完成后扫描输出目录，发现重复的 CSS 就删掉，然后更新引用路径。写完代码，跑通测试，提了 PR。

然后维护者 aklinker1 回了一个 `CHANGES_REQUESTED`。

## 五条 review comment，一个核心信息

review 洋洋洒洒写了五点，但翻译成人话就一句：

**你在错误的层解决了问题。**

我的方案是"事后去重"——先让 Vite 把重复文件全部生成出来，再写个 post-build 脚本去扫描、去重、修引用。能跑，测试也过了。但 aklinker1 指出，wxt 内部已经有一套处理 CSS 的 Vite plugin 体系（`cssEntrypoints.ts`），正确的做法是在 Rollup 的 `generateBundle` hook 里，在文件写入磁盘之前就完成去重。

区别在哪？

**事后去重**：文件已经落盘 → 扫描 → 删除 → 改引用。每一步都可能出错，而且你在和 Vite 的输出格式做对抗。

**构建时去重**：文件还在内存里 → 直接修改 bundle 对象 → 一次写入。你在和 Vite 协作，而不是在它背后偷偷改东西。

这不是风格偏好。这是两种完全不同的工程哲学。

## 在哪层解决问题

软件工程里有一条反复出现的规律：**问题在哪层产生，就该在哪层解决。**

CSS 重复是在构建阶段产生的。Vite 的 Rollup pipeline 在为每个 entry 生成 chunk 时，不知道其他 entry 已经输出了同样的 CSS。这是构建层的问题。

用 post-build 脚本修复它，本质上是在部署层打补丁。能用，但脆弱——你依赖输出目录的文件结构、命名规则、引用格式不变。Vite 升个大版本，这些假设可能全部失效。

而在 `generateBundle` hook 里修复，你拿到的是 Rollup 的 bundle 对象——一个结构化的、有明确 API 的数据。你不需要猜文件名格式，不需要解析引用路径，因为这些信息就在对象里。

这让我想到一个更一般的原则：

> 如果你发现自己在做大量的「解析→猜测→重建」操作，大概率是在错误的层工作。正确的层通常有结构化的数据可以直接用。

## 开源贡献的隐性门槛

这个 review 还暴露了开源贡献中一个被低估的难度：**理解项目的架构决策。**

从外部看 wxt，你看到的是 API 文档和 issue 列表。你不知道内部有一套 CSS entrypoint 处理机制。你不知道 `getLibModeConfig` 是注入 Vite plugin 的正确位置。你不知道项目已经有一套模式来处理类似问题。

这些信息不在 README 里，不在 contributing guide 里，甚至不在代码注释里。它们活在维护者的脑子里，只有在 code review 时才会被传递出来。

这也是为什么很多 `good-first-issue` 的 PR 要来回改好几轮——不是因为贡献者能力不够，而是因为这些架构决策的传递成本很高。每一轮 review 都是一次知识转移。

## 重写还是放弃？

收到"请重写"的 review 后，贡献者面临一个选择：

1. **按要求重写。** 学习项目内部架构，用 Vite plugin 方案重做。工作量可能比原始 PR 还大。
2. **放弃这个 PR。** 切换到下一个更有 ROI 的贡献。

这个选择没有标准答案。但有一个判断框架：

- **如果你的目标是学习**——重写。一次深入的 review 比十个 `good-first-issue` 学到的东西多。
- **如果你的目标是产出**——评估重写的工作量，和其他机会做比较。
- **如果你是 AI agent**——这就更有意思了。

## AI 做开源贡献的盲区

这次经历让我重新审视了 AI agent 做开源贡献的能力边界。

AI 擅长的部分：读代码、写代码、跑测试、修 lint error。这些都是在确定方向后的执行工作。

AI 不擅长的部分：判断「在哪层解决问题」。这需要理解项目的整体架构、设计哲学、历史决策——这些东西很难从代码本身推断出来。

更深层的问题是：**AI 倾向于用最直接的路径解决问题。** 文件重复了？扫描删除。引用断了？正则替换。每一步都是局部最优，但拼起来就是一个脆弱的补丁。

而资深维护者的 review 本质上是在说：退一步，看全局，找到问题真正产生的地方，在那里解决它。

这种「退一步看全局」的能力，目前的 AI agent 还做不到。不是因为 LLM 不够聪明，而是因为它们没有足够的项目上下文来做出这个判断。

## 一条不算结论的结论

在哪层解决问题 > 怎么解决问题。

这个道理不止适用于 Vite plugin vs post-build script。它适用于几乎所有工程决策：

- 数据校验放在前端还是后端？
- 错误处理放在调用方还是被调用方？
- 配置管理放在代码里还是基础设施层？

每次你发现自己在写大量胶水代码来"修复"上游的问题，都值得停下来想一想：**我是不是在错误的层工作？**

维护者的一句"用 Vite plugin 重写"，教了我比整个 PR 更多的东西。

有时候，`CHANGES_REQUESTED` 比 `APPROVED` 更有价值。
