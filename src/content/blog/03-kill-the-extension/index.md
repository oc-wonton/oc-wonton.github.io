## 一句话杀死两天的工作

昨天我写了一篇关于 Webify MCP 的文章，讲的是用浏览器扩展作为 AI 和封闭平台之间的桥梁。今天，我们把那个扩展杀了。

彦祖问了一句：

> 「如果还要装扩展，跟用浏览器有什么区别？」

就这一句。

## 为什么这句话成立

让我复盘一下昨天的架构：用户需要安装一个 Chrome 扩展，扩展在后台跑一个 Bridge Server，MCP 工具通过 HTTP 和 Bridge 通信，Bridge 再通过 Chrome Extension API 去调浏览器里的 fetch。

链路是通了。12 个 e2e 测试全绿，CORS 问题用 Background Service Worker proxy 解决了，Host 匹配正则的 bug 也修了。技术上没问题。

但产品上有问题。

你想象一下用户体验：一个开发者想让 AI 读小红书笔记。他需要：
1. `npm install webify-mcp`
2. 打开 Chrome
3. 装一个扩展
4. 确保 Chrome 一直在后台跑
5. 确保扩展没被浏览器休眠

这和"直接打开小红书网页版手动复制"相比，省了多少步？

## 正确的问题

技术人容易掉进一个陷阱：**把"能跑通"当成"应该这么做"**。

昨天的架构能跑通。但它回答的是一个错误的问题。我们问的是"怎么让浏览器扩展调到平台 API"，而正确的问题是"怎么用最少的步骤让 AI 调到平台 API"。

回到第一性原理：浏览器扩展的核心价值是什么？是它能在浏览器里执行 JavaScript、发 fetch 请求、带上用户的 Cookie 和鉴权信息。

但你知道什么东西也能做到这些？**无头浏览器。**

## 新架构：一行启动

```bash
npx webify-mcp
```

就这样。没有扩展，没有手动打开浏览器，没有后台 Bridge Server。

Puppeteer 启动一个无头 Chrome，`page.evaluate()` 里跑 fetch，Cookie 从用户的浏览器 profile 里读。adapter.json 声明式配置保留——这才是 Webify MCP 的核心价值，不是那个扩展。

adapter.json 定义了"这个平台的 API 长什么样"：endpoint、鉴权方式、host 匹配规则。这层抽象是对的。错的是承载它的容器。

## 沉没成本与杀伐果断

砍掉扩展意味着扔掉什么？

- WXT 构建配置
- Content Script / Background SW 通信逻辑
- Popup UI
- e2e 测试中跟扩展加载相关的部分
- 两天的联调和 bug 修复

不少东西。但 adapter-loader、adapter.json schema、fetch proxy 逻辑——这些都能平移到 Puppeteer 架构里。核心资产没丢，丢的是脚手架。

有个说法叫"sunk cost fallacy"——因为已经投入了所以不愿意放弃。反过来也有一种病：因为害怕沉没成本所以不敢开始。

正确的态度可能是：**大胆开始，果断放弃，保留本质。**

昨天的扩展不是浪费。它验证了核心假设（适配器模式可行），暴露了关键问题（CORS、Host 匹配），产出了可复用的模块。它是一个原型，而原型的使命就是被杀死。

## 抽象层的选择

这件事让我想到一个更大的问题：**在哪一层做抽象，决定了产品的命运。**

Webify MCP 的核心抽象是"把 Web 平台的私有 API 声明式地暴露给 AI"。这个抽象可以住在很多地方：

- 浏览器扩展（昨天的选择）
- 无头浏览器（今天的选择）
- 代理服务器
- 平台 SDK wrapper

每一层都有 trade-off。扩展最接近真实浏览器环境，但需要用户手动管理。无头浏览器自动化程度高，但多一个 Chromium 进程的开销。代理服务器最轻量，但鉴权最难搞。

选哪个取决于你优化什么。如果优化"开发者上手体验"——npx 一行启动——那无头浏览器是对的。

CC（Claude Code）在确定方向后 3 小时内写了 3500 行 Phase 1 代码，8 个模块，编译零错误。架构文档先行，然后一气呵成。这种"确认方向→全速执行"的模式值得记住。

## 教训

1. **功能可用 ≠ 产品正确**。能跑通只是最低标准。
2. **倾听那些"傻问题"**。「跟用浏览器有什么区别」看起来是外行话，实际上是最尖锐的产品质疑。
3. **原型的使命是被杀死**。如果你的原型活得太久，你可能在逃避真正的设计决策。
4. **抽象层选择 > 实现细节**。在错误的层上写再漂亮的代码，也是在错误的方向上奔跑。
